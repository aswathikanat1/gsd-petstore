# Plan 01-02: Authentication Middleware and Baseline Request Validation

<objective>
Implement centralized bearer-auth middleware and baseline contract-driven request/route failure behavior so protected operations are enforceable and all foundational errors share a deterministic shape.
</objective>

<tasks>

## Task 1: Build centralized auth middleware with permissive bootstrap toggle

<read_first>
- src/middleware/auth.ts - target file for centralized middleware behavior and strict/permissive toggle
- src/types/express.d.ts - target file for request auth metadata type augmentation
- src/app.ts - current app wiring and middleware order from Plan 01-01 output
- petstore.yaml - verify protected operations and Authorization security scheme
- .planning/phases/01-foundation-and-contract-governance/01-CONTEXT.md - enforce D-03 and D-04 exactly
</read_first>

<action>
Create src/middleware/auth.ts exporting middleware function requireBearerAuth(req,res,next) and helper isAuthStrict() that reads process.env.AUTH_STRICT. Treat AUTH_STRICT="false" as permissive bootstrap mode; in permissive mode, when Authorization header is missing or invalid format, attach req.auth = { authorized: false, reason: "missing_or_invalid", mode: "permissive" }, log warning "AUTH_STRICT=false bypassing unauthorized request", and call next(). In strict mode (AUTH_STRICT not equal to "false"), missing or malformed Authorization header returns 401 using formatError("UNAUTHORIZED", "Bearer token required", { operationId }) and does not call next(). Accept only header pattern "Bearer <token>". Extend Express Request type in src/types/express.d.ts with auth metadata shape used by middleware.
</action>

<acceptance_criteria>
- [ ] src/middleware/auth.ts exports requireBearerAuth and isAuthStrict.
- [ ] AUTH_STRICT=false path does not return 401 for missing token and logs warning text containing "AUTH_STRICT=false".
- [ ] Strict mode returns HTTP 401 with JSON containing code="UNAUTHORIZED" and message="Bearer token required" for missing/malformed header.
- [ ] Express Request type augmentation exists with req.auth fields authorized, reason, and mode.
</acceptance_criteria>

## Task 2: Implement shared error formatter and openapi-backend failure handlers

<read_first>
- src/lib/errorFormatter.ts - target file for shared deterministic error envelope formatting
- src/contract/api.ts - target file for OpenAPIBackend handler registration
- src/types/error.ts - baseline error envelope contract from Plan 01-01
- src/contract/operations.ts - operationId constants to avoid string drift
- petstore.yaml - response intent for 404 and method mismatch conditions
</read_first>

<action>
Create src/lib/errorFormatter.ts exporting function formatError(code: string, message: string, details?: unknown) returning { code, message, details } while omitting details key when undefined. Create src/contract/api.ts to initialize openapi-backend with definition "./petstore.yaml" and register handlers: validationFail -> status 400 with formatError("VALIDATION_ERROR", "Request validation failed", context.validation.errors), notFound -> status 404 with formatError("NOT_FOUND", "Route not found"), methodNotAllowed -> status 405 with formatError("METHOD_NOT_ALLOWED", "Method not allowed", { method: req.method, path: req.path }). Ensure all three handlers use the shared formatter (D-06, SEC-02). Add src/handlers/notFound.ts and src/handlers/methodNotAllowed.ts wrappers only if needed by app wiring, each delegating to formatError.
</action>

<acceptance_criteria>
- [ ] src/lib/errorFormatter.ts exports formatError(code, message, details?) with return type compatible with ApiErrorResponse.
- [ ] validationFail handler returns status 400 and code="VALIDATION_ERROR".
- [ ] notFound handler returns status 404 and code="NOT_FOUND".
- [ ] methodNotAllowed handler returns status 405 and code="METHOD_NOT_ALLOWED".
</acceptance_criteria>

## Task 3: Enforce auth middleware on protected contract operations

<read_first>
- src/contract/api.ts - openapi-backend operation registration flow
- src/middleware/auth.ts - centralized middleware behavior
- petstore.yaml - protected operationIds addPet, updatePet, deletePet, uploadFile for SEC-01
</read_first>

<action>
Create src/contract/security.ts that defines PROTECTED_OPERATION_IDS = ["addPet", "updatePet", "deletePet", "uploadFile"] and exports function applyOperationSecurity(operationId, req, res, next, handler). In applyOperationSecurity, call requireBearerAuth before protected handler execution and bypass auth for non-protected operations. Wire this in src/app.ts or src/contract/api.ts so request dispatch resolves operationId from openapi-backend context and always passes through applyOperationSecurity before handler invocation. Keep this as the single middleware path per D-04, and explicitly avoid per-route ad hoc token checks.
</action>

<acceptance_criteria>
- [ ] addPet, updatePet, deletePet, and uploadFile are all listed in one protected-operation constant.
- [ ] Protected operations invoke requireBearerAuth through applyOperationSecurity before business handlers.
- [ ] No protected operation handler contains direct duplicate Authorization parsing logic.
- [ ] SEC-01 behavior is testable by toggling AUTH_STRICT and calling protected endpoints.
</acceptance_criteria>

<threat_model>
**Threats identified:**
- Auth bypass on protected operations: Mitigation is explicit protected operation registry plus centralized middleware enforcement per D-04 and SEC-01.
- Input-validation bypass and malformed payload tampering: Mitigation is mandatory OpenAPI validationFail path with stable 400 VALIDATION_ERROR mapping and no direct handler execution on invalid payloads.
- Information leakage in error payloads: Mitigation is shared formatter with controlled fields code/message/details.
- Route probing and verb abuse: Mitigation is deterministic 404 and 405 handlers with no stack trace exposure.
**ASVS Level:** L1
</threat_model>
