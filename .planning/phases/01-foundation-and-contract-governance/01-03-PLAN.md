# Plan 01-03: Documentation Generation and Developer Usage Guide

<objective>
Deliver a repeatable docs pipeline that serves and generates API reference material directly from petstore.yaml, and provide a developer guide that standardizes local usage with auth and validation behaviors.
</objective>

<tasks>

## Task 1: Add runtime docs route sourced from petstore.yaml

<read_first>
- src/app.ts - existing middleware and route mount order from prior plans
- petstore.yaml - source contract file that must drive docs output
- .planning/phases/01-foundation-and-contract-governance/01-RESEARCH.md - redoc-express preference guidance
</read_first>

<action>
Install redoc-express and yamljs dependencies. Create src/docs/docsRouter.ts exporting function mountDocs(app) that loads petstore.yaml and mounts GET /api-docs via redoc-express with title "Swagger Petstore API" and specUrl "/petstore.yaml" (or equivalent in-memory spec injection). Update src/app.ts to call mountDocs(app) after core middleware setup so docs endpoint is always available in dev and CI smoke checks. Ensure docs route does not include manual markdown-generated endpoint descriptions; all operation rendering must come from petstore.yaml per D-07.
</action>

<acceptance_criteria>
- [ ] GET /api-docs returns HTTP 200 and renders Redoc UI.
- [ ] Docs UI content resolves from petstore.yaml, not hardcoded route text.
- [ ] src/app.ts mounts docs route exactly at /api-docs.
- [ ] No manual OpenAPI duplication file is introduced for docs rendering.
</acceptance_criteria>

## Task 2: Implement docs generation command and CI automation

<read_first>
- package.json - existing scripts from Plan 01-01
- .github/workflows/validate.yml - existing contract validation gate
- petstore.yaml - input file for lint and docs build commands
</read_first>

<action>
Add package.json script "docs:generate": "redocly lint petstore.yaml && redocly build-docs petstore.yaml --output docs/api-reference.html". Update .github/workflows/validate.yml to include docs generation step, or create .github/workflows/docs.yml that runs on pull_request and push to main with npm ci and npm run docs:generate. Keep command deterministic with exact output path docs/api-reference.html so CI can archive or diff artifacts. This fulfills OPS-01 and D-08 by ensuring docs are built from source spec without manual authoring.
</action>

<acceptance_criteria>
- [ ] package.json contains docs:generate script with both redocly lint petstore.yaml and redocly build-docs petstore.yaml --output docs/api-reference.html.
- [ ] CI workflow executes npm run docs:generate on pull_request.
- [ ] Running npm run docs:generate creates docs/api-reference.html.
- [ ] docs generation fails when petstore.yaml contains linting errors.
</acceptance_criteria>

## Task 3: Write developer usage guide for install, run, auth, and docs access

<read_first>
- docs/getting-started.md - create or update target developer guide
- package.json - authoritative command names to document
- src/middleware/auth.ts - AUTH_STRICT behavior that must be documented accurately
</read_first>

<action>
Create docs/getting-started.md with sections and exact headings: "Prerequisites", "Install", "Run Locally", "Authentication Mode", "Validate OpenAPI Contract", "Generate API Docs", and "View API Docs". Document commands npm ci, npm run dev, npm run openapi:lint, npm run docs:generate. Include auth behavior statement: "Set AUTH_STRICT=false for permissive bootstrap mode; omit or set AUTH_STRICT=true to enforce 401 for missing/invalid bearer token on protected operations." Include docs URL http://localhost:3000/api-docs.
</action>

<acceptance_criteria>
- [ ] docs/getting-started.md contains all required headings listed in this task.
- [ ] Guide includes exact commands npm ci, npm run dev, npm run openapi:lint, npm run docs:generate.
- [ ] Guide includes explicit AUTH_STRICT=false permissive behavior and strict-mode 401 behavior.
- [ ] Guide states docs URL as http://localhost:3000/api-docs.
</acceptance_criteria>

<threat_model>
**Threats identified:**
- Bearer-auth misunderstanding in developer usage leading to insecure local testing assumptions: Mitigation is explicit AUTH_STRICT guidance and protected-route expectations in getting-started docs.
- Input-validation misunderstanding causing clients to bypass schema expectations: Mitigation is documentation that validation is contract-driven from petstore.yaml and linked validation command in onboarding guide.
- Documentation drift from implementation contract: Mitigation is generating docs exclusively from petstore.yaml and running docs generation in CI.
- Exposure of stale docs in developer workflows: Mitigation is deterministic docs output command and automated regeneration on PRs.
- Misconfiguration of auth expectations by developers: Mitigation is explicit AUTH_STRICT guidance in docs/getting-started.md.
**ASVS Level:** L1
</threat_model>
