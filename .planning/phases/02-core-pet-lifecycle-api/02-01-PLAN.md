# Plan 02-01: Persistence Model, Create, and Get-by-ID Handlers

<objective>
Establish the typed pet repository and the create/fetch path for phase 2 so the API can persist pets in-memory, return created records, and resolve pets by ID with contract-compliant not-found handling.
</objective>

<tasks>

## Task 1: Introduce a typed pet repository and entity model

<read_first>
- src/contract/operations.ts - confirm addPet and getPetById operationId constants
- src/types/error.ts - preserve the shared error envelope baseline from phase 1
- petstore.yaml - validate Pet schema and response status intent
- .planning/phases/02-core-pet-lifecycle-api/02-CONTEXT.md - D-01 through D-07
</read_first>

<action>
Create src/types/pet.ts for the typed Pet entity shape used by the phase and create src/repositories/petRepository.ts with an in-memory store that can assign IDs, store normalized pet records, and retrieve records by numeric ID. Keep the repository interface small and testable so later plans can reuse it without reworking state semantics.
</action>

<acceptance_criteria>
- [ ] src/types/pet.ts defines a typed Pet model compatible with petstore.yaml.
- [ ] src/repositories/petRepository.ts supports create and findById operations.
- [ ] The repository assigns or preserves numeric IDs in a deterministic way.
- [ ] No external persistence dependency is introduced in this plan.
</acceptance_criteria>

## Task 2: Implement addPet and getPetById handlers

<read_first>
- petstore.yaml - inspect addPet and getPetById request/response bodies
- src/repositories/petRepository.ts - target repository contract for CRUD operations
- src/contract/api.ts - current OpenAPI backend dispatch wiring from phase 1
</read_first>

<action>
Create src/handlers/addPet.ts and src/handlers/getPetById.ts so addPet stores a new pet and returns the created payload, while getPetById resolves a single pet by ID or delegates a not-found response to the shared error pipeline. Wire both handlers into the OpenAPI dispatch layer without duplicating route logic.
</action>

<acceptance_criteria>
- [ ] addPet returns a created pet payload that matches the contract shape.
- [ ] getPetById returns the requested pet when the ID exists.
- [ ] getPetById returns a contract-compliant 404 response when the ID is missing.
- [ ] Handler code does not duplicate repository logic inline.
</acceptance_criteria>

</tasks>

<verification>

2. Request a nonexistent pet ID and confirm the shared not-found response is returned.
3. Run the relevant unit or endpoint tests for create/read behavior once added in phase 2 plan 03.
</verification>

<success_criteria>
- [ ] The API can create pets and fetch them by ID using a small, typed repository boundary.
- [ ] Contract-compliant not-found handling is preserved for missing pet IDs.
- [ ] The phase 2 core data model is ready for update and delete work.
</success_criteria>

<must_haves>
- Create and read operations stay aligned to petstore.yaml.
- Repository semantics remain in-memory for this phase.
- Error handling continues to use the phase 1 shared formatter.
- The plan leaves room for later update/delete and test coverage without refactoring the core model.
</must_haves>

<threat_model>
**Threats identified:**
- ID drift between stored pets and path parameters: mitigate by making the repository the single source of ID assignment and lookup.
- Contract mismatch in create/read payloads: mitigate by typing the Pet model from the OpenAPI schema and keeping handlers thin.
- Hidden state coupling across handlers: mitigate by centralizing all storage operations behind one repository interface.
- Missing-pet behavior accidentally degrading into generic errors: mitigate by routing not-found through the shared error pipeline.
**ASVS Level:** L1
</threat_model>
