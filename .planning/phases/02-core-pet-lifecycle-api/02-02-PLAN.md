# Plan 02-02: Update and Delete Handlers with Contract-Compliant Responses

<objective>
Extend the phase 2 repository and handler layer to support full pet replacement updates and deterministic deletions, preserving OpenAPI response semantics and the shared error envelope behavior.
</objective>

<tasks>

## Task 1: Extend repository support for update and delete operations

<read_first>
- src/repositories/petRepository.ts - extend the repository contract established in plan 02-01
- petstore.yaml - confirm updatePet and deletePet response semantics
- .planning/phases/02-core-pet-lifecycle-api/02-CONTEXT.md - D-03 through D-07
</read_first>

<action>
Add update and delete methods to the in-memory pet repository so an existing pet can be replaced by ID and removed deterministically. Keep ID ownership server-side and make missing-ID behavior explicit so handler code can map it to the shared not-found response path.
</action>

<acceptance_criteria>
- [ ] Repository update replaces the stored pet at the given ID.
- [ ] Repository delete removes the record and returns a deterministic outcome.
- [ ] Missing records are distinguishable from successful mutations.
- [ ] The repository interface remains narrow and easy to unit test.
</acceptance_criteria>

## Task 2: Implement updatePet and deletePet handlers

<read_first>
- petstore.yaml - inspect updatePet and deletePet request/response semantics
- src/repositories/petRepository.ts - target mutation operations for handlers
- src/lib/errorFormatter.ts - shared error output for failure paths
</read_first>

<action>
Create src/handlers/updatePet.ts and src/handlers/deletePet.ts so updatePet fully replaces the record associated with the path ID and deletePet removes the record associated with the path ID. Ensure both handlers delegate missing-record and validation-adjacent failures through the shared formatter rather than improvising new error shapes.
</action>

<acceptance_criteria>
- [ ] updatePet returns the updated Pet payload for an existing record.
- [ ] deletePet returns the contract-approved deterministic confirmation response.
- [ ] Missing IDs surface a contract-compliant 404 response.
- [ ] No ad hoc error envelope shape is introduced.
</acceptance_criteria>

</tasks>

<verification>
1. Update an existing pet and confirm the stored record changes.
2. Delete an existing pet and confirm it can no longer be fetched.
3. Exercise the missing-ID branches for both handlers and confirm 404 behavior.
</verification>

<success_criteria>
- [ ] The API can update and delete pets using the same in-memory repository from plan 02-01.
- [ ] Update and delete responses remain deterministic and contract-aligned.
- [ ] No route-specific persistence logic leaks outside the repository boundary.
</success_criteria>

<must_haves>
- Update is a full replacement, not a partial merge.
- Delete removes data and returns a consistent response.
- Error handling continues to use the phase 1 shared formatter.
- Phase 2 CRUD semantics stay independent of later search and upload behavior.
</must_haves>

<threat_model>
**Threats identified:**
- Partial updates accidentally mutating only some fields: mitigate by treating updatePet as full resource replacement.
- Delete paths returning inconsistent success payloads: mitigate by using one deterministic confirmation shape.
- Missing-record handling becoming inconsistent between read, update, and delete: mitigate by mapping all absent-ID cases through the shared error pipeline.
- Repository mutation bugs affecting later tests: mitigate by keeping state changes isolated behind a narrow repository interface.
**ASVS Level:** L1
</threat_model>
