# Plan 02-03: Endpoint-Level Tests for Happy and Failure CRUD Scenarios

<objective>
Add end-to-end tests for the phase 2 lifecycle operations so create, fetch, update, and delete behavior is exercised against the contract rather than only by unit-level assumptions.
</objective>

<tasks>

## Task 1: Build a lifecycle-focused test harness

<read_first>
- src/handlers/addPet.ts - create behavior to validate in tests
- src/handlers/getPetById.ts - read behavior to validate in tests
- src/handlers/updatePet.ts - update behavior to validate in tests
- src/handlers/deletePet.ts - delete behavior to validate in tests
- src/repositories/petRepository.ts - resettable state boundary for test fixtures
</read_first>

<action>
Create a targeted test harness that can bootstrap the phase 2 handlers and repository with predictable fixture state. Keep the harness lightweight so endpoint-level coverage reflects the actual handler flow instead of bypassing it with overly mocked internals.
</action>

<acceptance_criteria>
- [ ] Tests can initialize a clean repository state for each scenario.
- [ ] Handler behavior is exercised through the real phase 2 wiring.
- [ ] The harness stays small and does not introduce extra production behavior.
</acceptance_criteria>

## Task 2: Cover happy-path and failure CRUD scenarios

<read_first>
- petstore.yaml - confirm success and error status codes for each lifecycle operation
- src/lib/errorFormatter.ts - shared error output shape to assert in failures
- .planning/REQUIREMENTS.md - PET-01 through PET-04 requirements to satisfy
</read_first>

<action>
Write test cases that verify create returns a pet payload, get returns the stored pet or 404 when missing, update replaces the record, and delete removes the record with a deterministic success response. Include at least one malformed or missing-ID failure case per operation group where the contract expects it.
</action>

<acceptance_criteria>
- [ ] create, get, update, and delete happy paths are covered.
- [ ] Missing-resource behavior is covered with contract-compliant 404 assertions.
- [ ] Response bodies and status codes are asserted, not just handler invocation counts.
- [ ] The suite protects the lifecycle semantics from regressions.
</acceptance_criteria>

</tasks>

<verification>
1. Run the lifecycle test suite and confirm the create/read/update/delete scenarios pass.
2. Confirm missing-ID and malformed-request assertions fail when contract behavior is broken.
3. Review coverage to ensure each core lifecycle endpoint is exercised at least once on a real handler path.
</verification>

<success_criteria>
- [ ] The phase 2 core lifecycle contract is covered by repeatable endpoint-level tests.
- [ ] Happy-path and failure-path behavior are both validated against the OpenAPI semantics.
- [ ] The test suite provides confidence for moving to phase 3 discovery and media work.
</success_criteria>

<must_haves>
- Tests exercise the actual lifecycle flow, not just isolated pure functions.
- Contract behavior remains the source of truth for expected outcomes.
- The suite covers both success and failure paths for the core CRUD surface.
- Coverage is focused on PET-01 through PET-04 and does not drift into phase 3 scope.
</must_haves>

<threat_model>
**Threats identified:**
- Silent contract regressions in core CRUD endpoints: mitigate with endpoint-level assertions on both status codes and payloads.
- Overmocking that hides integration bugs: mitigate by keeping the repository/handler wiring real in the harness.
- Missing-ID and malformed-input regressions escaping detection: mitigate with explicit negative test cases for each lifecycle branch.
- Test brittleness from shared state leakage: mitigate with clean repository reset behavior per scenario.
**ASVS Level:** L1
</threat_model>
