# Plan 03-01: Status Filtering Endpoint with Enum Validation and Tests

<objective>
Implement and verify status-based pet discovery so clients can filter pets by allowed status values with deterministic success and failure semantics.
</objective>

<tasks>

## Task 1: Extend repository for status filtering

<read_first>
- src/repositories/petRepository.ts - existing in-memory lifecycle boundary from phase 2
- src/types/pet.ts - status field type constraints
- petstore.yaml - findPetsByStatus query semantics and response intent
- .planning/phases/03-discovery-and-media-workflows/03-CONTEXT.md - D-01, D-02, D-03
</read_first>

<action>
Add repository support to retrieve pets by one or more status values while preserving in-memory semantics. Ensure filtering returns a deterministic array and does not mutate stored records.
</action>

<acceptance_criteria>
- [ ] Repository exposes a filter method for status-based lookup.
- [ ] Filtering supports comma-separated or repeated status query values as contract intent.
- [ ] Valid status values return an array (including empty array when no matches).
- [ ] Filtering logic does not mutate repository state.
</acceptance_criteria>

## Task 2: Implement findPetsByStatus handler and wire into contract API

<read_first>
- petstore.yaml - operationId findPetsByStatus and allowed status enum values
- src/contract/api.ts - current operation registration table
- src/lib/errorFormatter.ts - shared deterministic error output
</read_first>

<action>
Create src/handlers/findPetsByStatus.ts to parse status query parameters, validate against available/pending/sold, and return matching pets. Return 400 with shared error formatter for invalid status input. Wire the handler to the findPetsByStatus operation in src/contract/api.ts.
</action>

<acceptance_criteria>
- [ ] findPetsByStatus returns 200 with an array of pets for valid status input.
- [ ] Invalid status input returns 400 with deterministic error payload.
- [ ] Handler is registered for operationId findPetsByStatus in contract API.
- [ ] No phase-2 CRUD behavior regresses due to repository changes.
</acceptance_criteria>

</tasks>

<verification>
1. Query findPetsByStatus with available/pending/sold and confirm filtered results.
2. Query with an invalid status value and confirm 400 invalid-status response.
3. Re-run core CRUD tests to ensure repository extension did not regress phase 2 behavior.
</verification>

<success_criteria>
- [ ] Users can discover pets by status with predictable filtering behavior.
- [ ] Invalid status input is explicitly rejected and consistently formatted.
- [ ] Discovery behavior remains contract-aligned and test-covered.
</success_criteria>

<must_haves>
- Allowed enum values are enforced at handler boundary.
- Response for valid input is always a JSON array.
- Error handling uses shared formatter and stable status mapping.
- Repository remains the only source of pet-query state.
</must_haves>

<threat_model>
**Threats identified:**
- Invalid filter values silently accepted and causing undefined behavior: mitigate with explicit enum validation and 400 responses.
- Filter logic mutating shared state: mitigate with read-only repository filtering methods.
- Contract drift from query parsing differences: mitigate by aligning parsing behavior to petstore.yaml semantics.
- Regression in CRUD flows due to repository changes: mitigate with regression tests across lifecycle operations.
**ASVS Level:** L1
</threat_model>
