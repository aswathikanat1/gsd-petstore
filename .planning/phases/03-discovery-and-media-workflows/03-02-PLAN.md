# Plan 03-02: Form-Style Update Endpoint and Backward Compatibility Checks

<objective>
Implement query/form-based pet updates so legacy-style integrations can update name/status while preserving phase 2 full JSON update semantics.
</objective>

<tasks>

## Task 1: Add partial-merge repository support for form updates

<read_first>
- src/repositories/petRepository.ts - existing create/read/update/delete methods
- src/types/pet.ts - mutable fields allowed by form updates
- .planning/phases/03-discovery-and-media-workflows/03-CONTEXT.md - D-04, D-05, D-06
</read_first>

<action>
Add repository support for partial updates that merge only provided fields (name/status) into an existing pet record while preserving all other values. Keep this path explicit so it does not alter full-replacement updatePet behavior.
</action>

<acceptance_criteria>
- [ ] Repository exposes a partial update method for form-style updates.
- [ ] Only provided fields are modified; omitted fields are preserved.
- [ ] Missing pet IDs are handled explicitly for 404 mapping.
- [ ] Existing full update path remains unchanged.
</acceptance_criteria>

## Task 2: Implement updatePetWithForm handler and compatibility tests

<read_first>
- petstore.yaml - updatePetWithForm path/query parameter contract
- src/contract/api.ts - operation registration map
- src/lib/errorFormatter.ts - deterministic error payload behavior
</read_first>

<action>
Create src/handlers/updatePetWithForm.ts that parses petId from path plus optional name/status inputs, applies partial merge update, and returns updated Pet payload. Return 400 for invalid input and 404 for missing pets via shared formatter. Wire the handler into src/contract/api.ts and add tests that prove form updates and JSON updates coexist without semantic drift.
</action>

<acceptance_criteria>
- [ ] updatePetWithForm returns 200 with updated pet payload for valid input.
- [ ] updatePetWithForm preserves untouched fields.
- [ ] Invalid input and missing resources return deterministic 400/404 responses.
- [ ] Backward compatibility checks confirm phase 2 updatePet still behaves as full replacement.
</acceptance_criteria>

</tasks>

<verification>

2. Update only status via form endpoint and confirm other fields are preserved.
3. Run phase 2 updatePet tests to verify full replacement semantics still hold.
4. Confirm missing pet ID returns contract-compliant 404 response.
</verification>

<success_criteria>
- [ ] Form-based pet updates work for integrations that rely on query-style updates.
- [ ] Partial merge behavior is predictable and test-covered.
- [ ] JSON full replacement update semantics remain backward compatible.
</success_criteria>

<must_haves>
- Form updates are partial merge only.
- JSON updates remain full replacement behavior.
- Error responses remain stable and formatter-driven.
- Implementation remains contract-first and operationId-bound.
</must_haves>

<threat_model>
**Threats identified:**
- Form updates accidentally replacing full resources: mitigate via dedicated partial-merge repository method.
- Backward incompatibility with existing JSON update behavior: mitigate with explicit compatibility tests.
- Invalid status/name inputs causing inconsistent state: mitigate with validation and shared 400 response mapping.
- Missing-resource paths diverging from other handlers: mitigate through shared 404 handling conventions.
**ASVS Level:** L1
</threat_model>
