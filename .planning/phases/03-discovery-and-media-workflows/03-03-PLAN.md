# Plan 03-03: Upload Endpoint with Binary Handling and Error Coverage

<objective>
Implement image-upload behavior for pets with deterministic response semantics so clients can attach binary media metadata and receive contract-compliant success and error responses.
</objective>

<tasks>

## Task 1: Extend repository and type model for upload metadata

<read_first>
- src/types/pet.ts - current pet model definition from phase 2
- src/repositories/petRepository.ts - state model and mutation boundaries
- .planning/phases/03-discovery-and-media-workflows/03-CONTEXT.md - D-07, D-08, D-09
</read_first>

<action>
Add a minimal upload metadata model linked to pets in memory (for example: filename/contentType/size/additionalMetadata markers). Ensure metadata writes are deterministic and do not alter unrelated pet fields.
</action>

<acceptance_criteria>
- [ ] Repository supports storing upload metadata associated with pet ID.
- [ ] Metadata updates preserve existing pet core fields.
- [ ] Missing pet IDs are surfaced explicitly for 404 mapping.
- [ ] No external file storage dependency is added.
</acceptance_criteria>

## Task 2: Implement uploadFile handler and contract mapping

<read_first>
- petstore.yaml - uploadFile requestBody and response codes (200/400/404)
- src/contract/api.ts - current operation registration table
- src/lib/errorFormatter.ts - shared error response behavior
</read_first>

<action>
Create src/handlers/uploadFile.ts that validates file presence, validates pet existence, records upload metadata in repository, and returns ApiResponse-compatible payload on success. Return 400 when no file is provided and 404 when pet does not exist. Wire uploadFile operation to the new handler in contract API.
</action>

<acceptance_criteria>
- [ ] uploadFile returns 200 with ApiResponse-shaped payload on success.
- [ ] Missing file returns 400 with deterministic error payload.
- [ ] Missing pet returns 404 with deterministic error payload.
- [ ] Handler is registered for operationId uploadFile.
</acceptance_criteria>

<acceptance_criteria>

</tasks>

<verification>

<success_criteria>
- [ ] Image uploads return consistent success and error responses.
- [ ] Upload behavior is test-covered for happy and failure paths.
- [ ] Implementation remains in-memory and contract-compliant for phase scope.
</success_criteria>

<must_haves>
- Success path uses ApiResponse fields from contract.
- Failure paths are deterministic and formatter-driven.
- Upload metadata integrates cleanly with existing repository design.
- No out-of-scope persistence infrastructure is introduced.
</must_haves>

<threat_model>
**Threats identified:**
- Missing file payload accepted silently: mitigate with explicit validation and 400 response.
- Upload metadata linked to wrong pet: mitigate with strict petId existence checks before write.
- Inconsistent success payload shape: mitigate by enforcing ApiResponse field structure in handler/tests.
- Upload features introducing persistence scope creep: mitigate by keeping metadata in-memory only.
**ASVS Level:** L1
</threat_model>
