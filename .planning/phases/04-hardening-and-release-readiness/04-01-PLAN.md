# Plan 04-01: Global Error Handling and Standardized Default Responses

<objective>
Harden runtime reliability by enforcing a single global fallback error path that maps unexpected failures to a deterministic, contract-compliant response envelope.
</objective>

<tasks>

## Task 1: Implement global fallback error contract in app boundary

<read_first>
- src/app.ts - existing express middleware chain and current error boundary
- src/lib/errorFormatter.ts - shared formatter used for deterministic response payloads
- src/types/error.ts - canonical error response shape
- .planning/phases/04-hardening-and-release-readiness/04-CONTEXT.md - D-01, D-02, D-03
</read_first>

<action>
Refine the global Express error middleware in src/app.ts so all uncaught exceptions map to a strict `INTERNAL_ERROR` contract envelope without stack leakage. Ensure optional diagnostic details are controlled and safe while preserving machine-consumable structure.
</action>

<acceptance_criteria>
- [ ] Uncaught runtime errors return deterministic 5xx payload with code/message/details shape.
- [ ] HTTP responses do not include stack traces or raw internal objects.
- [ ] Global error mapping is centralized in one middleware path.
- [ ] Existing successful and known 4xx operation paths are not altered.
</acceptance_criteria>

## Task 2: Preserve explicit contract-level 4xx behaviors and add regression checks

<read_first>
- src/contract/api.ts - validationFail, unauthorizedHandler, notFound, methodNotAllowed mappings
- test/petLifecycle.test.ts - existing endpoint behavior assertions
- petstore.yaml - default/known error response intent per operation
</read_first>

<action>
Verify and, if needed, harden contract API error mappings so validation/auth/not-found/method failures keep deterministic 4xx responses while unexpected errors flow to global fallback. Extend tests to cover at least one unexpected-failure scenario and assert fallback envelope consistency.
</action>

<acceptance_criteria>
- [ ] validation/auth/not-found/method-not-allowed mappings remain deterministic and unchanged.
- [ ] At least one unexpected-failure test asserts default 5xx error envelope behavior.
- [ ] No existing phase 1-3 endpoint contracts regress.
- [ ] SEC-03 behavior is demonstrably testable.
</acceptance_criteria>

</tasks>

<verification>
1. Trigger a controlled unexpected error and confirm deterministic 5xx `INTERNAL_ERROR` envelope.
2. Re-run key 4xx paths (validation/auth/not-found/method) and confirm existing response contracts.
3. Run build and tests to ensure hardening changes do not break phase 1-3 behavior.
</verification>

<success_criteria>
- [ ] Default/unexpected failures are captured and exposed through a consistent error contract.
- [ ] Error handling remains predictable across all operation classes.
- [ ] Regression coverage exists for fallback error behavior.
</success_criteria>

<must_haves>
- One global fallback error middleware path.
- Stable code/message/details envelope for 5xx fallback.
- No sensitive internal leakage in responses.
- Existing 4xx contract semantics remain intact.
</must_haves>

<threat_model>
**Threats identified:**
- Internal exception leakage revealing stack traces: mitigate via strict fallback envelope sanitization.
- Fragmented error handling creating inconsistent client behavior: mitigate via single global boundary and shared formatter.
- Regression of existing 4xx paths while hardening 5xx behavior: mitigate with targeted regression tests.
- Silent failures bypassing standardized mapping: mitigate by enforcing middleware ordering and centralized catch path.
**ASVS Level:** L1
</threat_model>
